Gemini 3.1 Pro's Thinking LevelsGoogle's new three-tier system lets you dial reasoning up or down per request

Every API call to a frontier model used to cost the same regardless of whether you were formatting a string or untangling a dependency graph. Gemini 3.1 Pro changes that with thinking levels, a simple parameter that controls how much reasoning the model applies to each request. You pick Low, Medium, or High, and you pay accordingly.

01
The Problem
Why most AI tokens go to waste
One price for everything

Before thinking levels, every request to a frontier model got the same heavy reasoning treatment. You ask it to format a JSON payload, it thinks for 10 seconds. You ask it to refactor an authentication module, same 10 seconds. The cost was flat regardless of complexity. There was no way to tell the model that a task was simple and it should just get it done quickly.

Where I noticed it

I was running agentic workflows where roughly 70% of the tasks were simple file reads and string formatting, but I was burning the same tokens as the 30% that actually needed deep reasoning. A typical pipeline might read a config file, extract a value, format it into a template, then do one genuinely hard step like analyzing a stack trace. Three cheap tasks subsidizing one expensive one, every single run.

02
How Thinking Levels Work
Low, Medium, and High
Low
Fast completions

Minimal internal reasoning. Good for formatting, simple transforms, and boilerplate generation. The model spends very little time in its chain of thought before producing output. Roughly equivalent cost to using a smaller model like Flash, but with Pro's broader knowledge and instruction following. Perfect for the boring 70% of any pipeline.

Medium
Balanced reasoning

Handles moderately complex tasks like code review, debugging straightforward issues, and writing tests. According to early adopters, this level covers about 80% of real coding tasks. It thinks enough to catch non-obvious bugs but does not spiral into over-analysis. This is the default you should start with for most work.

High
Full depth reasoning

Comparable to Gemini 3 Deep Think but in a production-ready package. This is for multi-step refactors, architectural decisions, and complex debugging across multiple files. The model takes its time, explores multiple approaches internally, and gives you its best answer. Reserve this for the tasks that actually justify the cost.

03
Real World Setup
Building a two-tier system
Combine Flash and Pro with thinking levels

The approach that worked best for me was using Flash for truly trivial tasks (reading files, simple string operations) and Pro with thinking levels for everything else. Flash handles the bottom tier at rock-bottom cost. Then for Pro requests, I start at Medium and only escalate to High when the first attempt fails or when the task clearly requires it, like refactoring a module that touches six other files. 

import google.genai as genai

client = genai.Client()

def call_gemini(prompt, thinking_level="medium"):
    response = client.models.generate_content(
        model="gemini-3.1-pro",
        contents=prompt,
        config={
            "thinking_config": {
                "thinking_level": thinking_level
            }
        }
    )
    return response.text

simple_task = call_gemini(
    "Format this as a TypeScript interface: name, age, email",
    thinking_level="low"
)

review_task = call_gemini(
    "Review this function for edge cases:\n" + code_snippet,
    thinking_level="medium"
)

hard_task = call_gemini(
    "Refactor the auth module to support OAuth2 and session tokens "
    "while maintaining backward compatibility with the existing API",
    thinking_level="high"
)
Agentic workflow tip

For agentic pipelines, use the customtools endpoint and set the thinking level per tool call. Your orchestrator can classify each step's complexity and route it to the right level automatically. Start everything at Medium, then build a simple retry mechanism that bumps to High on failure. Most tasks never need the retry.

04
What I Actually Saved
Token costs before and after
The experiment

I ran my coding agent pipeline for one week using flat Pro for every request. Then I ran the same pipeline for another week with thinking levels enabled. The workload was identical: processing pull requests, generating tests, reviewing code, and formatting outputs. Same repo, same types of tasks, same volume.

The breakdown

With thinking levels, the distribution settled at roughly 65% Low, 25% Medium, and 10% High. Total token spend dropped by about 3.5x compared to the flat Pro week. The quality on Low tasks was identical because those tasks never needed deep reasoning in the first place. Medium caught most bugs on the first pass. High was reserved for the gnarly cross-file dependency issues that actually benefited from extended reasoning.

Cost comparison
Before
Every request hit Pro at full reasoning cost. Simple formatting tasks, file reads, and template generation all consumed the same token budget as complex multi-file refactors. Weekly spend was consistent but high.
After
Low-tier tasks (65% of volume) cost a fraction of what they used to. Medium handled the bulk of real work. Only the genuinely hard 10% ran at full price. The net result was a 3.5x reduction in total spend with no measurable drop in output quality.
05
Where It Falls Short
Honest limitations
What to watch out for
1Choosing the right level is entirely on you. There is no auto-detection yet. If you set Low for something that needed High, you get garbage and have to retry. Sometimes that retry costs more than just using High in the first place.
2The Medium tier occasionally gets stuck in a reasoning loop on certain types of recursive problems. I hit this twice during my testing week, both times on functions with deeply nested callbacks. Bumping to High resolved it immediately.
3Latency on High is noticeably longer than Claude Opus for comparable tasks. In my testing, High was about 15-20% slower on average. If you are latency-sensitive, this matters.
4The cost savings assume you classify tasks correctly. If your orchestrator is too aggressive about using Low, you end up with more retries and the savings shrink. It takes some tuning to find the right thresholds for your specific workload.
06
The Bigger Picture
Why adjustable reasoning matters
Industry direction

This is not just about Gemini. It signals where the whole industry is heading. Instead of one-size-fits-all pricing for intelligence, we are moving toward granular compute allocation. OpenAI is rumored to be working on something similar for their reasoning models. Anthropic already offers different tiers through model selection, but per-request reasoning control within a single model is the next logical step.

What it changes for builders

For developers building AI-powered products, this changes the unit economics fundamentally. Features that would have been too expensive at flat-rate pricing become viable when you can route 65% of requests through the cheap tier. Think about a code review tool that uses Low for style checks, Medium for logic review, and High only for security audits. That product was not economically feasible before. Now it is.

Summary
Pay for the reasoning you actually need.

Thinking levels turn AI spending from a flat tax into a sliding scale. Start with Medium, drop to Low for the boring stuff, and save High for when it counts.
