Developer Tools

Portless Fixes LocalhostNo more memorizing which port runs what

You run your frontend on localhost:3000, your API on 8080, your database admin on 5432, and a background job processor on 4000. When you need to share your work, you fire up ngrok and hope you remembered the right port. Portless replaces this entire workflow with readable domain names like app.local and api.local, handles HTTPS automatically, and rewrites URLs on the fly so everything just works.

01
The localhost port problem
Why juggling ports is broken
Mental overhead

I work on three different projects each week. One uses 3000 for the frontend and 8000 for the API. Another uses 4000 and 3001. The third has five different services on ports I can never remember. Every time I switch contexts, I open my terminal, scroll back through history to find the right npm run command, or open a README to check which port is which. It is a tiny friction but it happens dozens of times a day.

HTTPS in development

Some browser APIs only work over HTTPS. Testing OAuth flows, service workers, or anything involving secure contexts means generating self signed certificates, ignoring browser warnings, and updating your localhost URLs to use https://. When you add multiple services that need to talk to each other, the certificate setup becomes a small project of its own.

Sharing work requires manual tunneling

When a designer asks to see your progress or you need to test on a mobile device, you fire up ngrok, copy the randomly generated URL, and share it. If your app has hardcoded localhost URLs, they break because the tunnel uses a public domain. You either rewrite your URLs manually or set up environment variables to handle both local and tunneled contexts. It works but it is clunky.

02
What Portless does
Virtual domains with automatic URL rewriting
Core feature
Local domain mapping

You create a config file that maps localhost:3000 to app.acme.local and localhost:8080 to api.acme.local. Portless runs as a background daemon and intercepts requests to those domains, routing them to the correct port automatically.

Automatic
Built in HTTPS

Portless generates and manages local SSL certificates so https://app.acme.local works without browser warnings. No mkcert, no manual cert installation, no ignoring security errors. It just works.

Smart
URL rewriting

When your app sends HTML, CSS, or JSON over the network, Portless automatically rewrites any localhost URLs to match the domain you are accessing. If you visit app.acme.local, references to localhost:8080 become api.acme.local automatically.

The workflow in practice

You define domains in portless.config.js, start the daemon once, and forget it exists. From that point forward, you type app.acme.local instead of localhost:3000. Your app sees requests coming from a real looking domain, HTTPS works, and URLs stay consistent whether you are testing locally or sharing via ngrok.

03
How it handles public sharing
Integrated ngrok tunneling
Automatic tunnel setup

Portless integrates with ngrok directly. You configure a public domain in your config file, run one command, and Portless exposes your local service with a stable URL. The same URL rewriting engine that works locally applies to the tunnel, so hardcoded localhost references transform into the public domain automatically.

Consistent URLs across contexts

Your code does not need to know whether it is running locally or publicly. Portless rewrites URLs based on the request host. If someone accesses your app via the ngrok URL, all embedded links and API calls use the public domain. If you access it via app.local, they use local domains. Same codebase, zero conditional logic.

04
How it compares to alternatives
The localhost tooling landscape
What other tools do
Ngrok
Only handles public tunneling. You still manage localhost ports manually, HTTPS locally requires separate setup, and URL rewriting is not automatic. Ngrok is great at what it does but it does not solve the local domain problem.
Localias
Provides custom local domain aliases but does not handle URL rewriting or tunneling. It solves the naming problem but you still need to set up HTTPS and manage public sharing separately.
Caddy
A reverse proxy that can handle local HTTPS and domain routing. More powerful than Portless but requires deeper configuration. It is a general purpose web server, not a dev specific tool. You gain flexibility but lose the zero config experience.
Where Portless fits

Portless combines local domain mapping, HTTPS, ngrok tunneling, and URL rewriting in one package. If you want readable local domains and do not want to configure a reverse proxy, it is the most complete solution. If you need advanced routing logic or run dozens of services, Caddy gives you more control.

05
Who benefits most
Ideal use cases
Fullstack developers

If you run a frontend and one or more backend services locally, Portless eliminates port confusion. You define domains once and stop thinking about numbers.

Teams sharing work often

Designers, product managers, or remote teammates who need to see in progress work benefit from stable ngrok URLs that do not break internal links.

Anyone testing HTTPS features

Service workers, web crypto APIs, or OAuth flows all require HTTPS. Portless gives you that without manual certificate management.

06
What this means for local dev
The broader shift toward better DX
Port numbers are an implementation detail

Portless treats localhost ports the way DNS treats IP addresses. You should not need to remember them. The tool maps human readable names to the underlying network details and stays out of your way. This is how local development should work.

Zero config is the goal

Developer tools are moving toward less manual setup. Vite auto opens your browser. Next.js handles routing without configuration. Portless follows that pattern: define your domains once and the rest happens automatically. The best tools make infrastructure invisible.

Takeaway
Localhost should feel like production

Portless makes local development feel like working with real domain names. No more memorizing ports, no more manual tunneling, no more broken URLs when you share your work. If you run multiple services locally, it is worth trying.
